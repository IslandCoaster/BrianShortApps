# Engineering Philosophy

## Purpose

The BrianShortApps engineering philosophy is to build deliberately, visibly, and in alignment with the platform architecture.

## Engineering Belief

A good system is not only functional.

It is understandable, diagnosable, maintainable, and extensible.

## Development Approach

Engineering work should move through this sequence:

1. Define the capability.
2. Register the capability.
3. Place it in the architecture.
4. Implement the resource.
5. Connect diagnostics.
6. Verify behavior.
7. Document the result.

## Bias Toward Clarity

Prefer explicit structure over clever shortcuts.

Prefer small durable abstractions over broad premature frameworks.

Prefer visible ownership over implicit tribal knowledge.

## Technical Debt Rule

Technical debt is not just bad code.

Technical debt is any implementation that makes the platform harder to understand, diagnose, secure, deploy, or extend.

## Platform-First Discipline

If a capability may reasonably serve more than one application, it should be evaluated as a platform capability before being implemented inside a product.

## Definition of Progress

Progress is not only a working feature.

Progress is a feature that fits the architecture, has ownership, can be diagnosed, and can be safely extended.
