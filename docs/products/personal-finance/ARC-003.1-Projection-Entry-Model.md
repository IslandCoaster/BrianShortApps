# BrianShortApps Projection Entry Model

## Document Status

| Field | Value |
|---|---|
| Architecture milestone | ARC-003.1 |
| Status | Accepted |
| Product | BrianShortApps Personal Finance |
| Document authority | Canonical planning-projection primitive |
| Parent architecture | ARC-003 — Financial Flow Model |

This document defines the canonical `ProjectionEntry` primitive used to
represent future, non-posted financial movements before they are replayed into
account-level projections.

---

# 1. Purpose

A Projection Entry represents a planned, expected, or simulated future movement
that may change an account balance.

Projection Entries are not financial history.

They exist to support:

- Future cash projections
- Planned debt settlements
- Account liquidity analysis
- Overdraft detection
- Transfer recommendations
- Scenario comparison
- Later reconciliation against posted activity

The purpose of this model is to prevent account projection logic from becoming
coupled directly to funding sources, debt accounts, obligations, rewards, or
institution-specific rules.

---

# 2. Core Principle

Domain engines produce Projection Entries.

Projection engines consume Projection Entries.

Canonical flow:

```text
Funding Source
→ Funding Deposit Projection
→ Projection Entry

Debt Account
→ Settlement Projection
→ Projection Entry

Future Transfer
→ Transfer Projection
→ Projection Entry

Projection Entries
→ Account Projection Replay
→ Projected Account State
```

The account projection engine must not need to understand why an entry exists.

It only needs to understand:

- Which account is affected
- When the movement occurs
- How much the balance changes
- How entries are ordered
- How the movement should be explained

# 3. Canonical Type

```ts
export type ProjectionEntryType =
  | "funding-deposit"
  | "planned-settlement"
  | "planned-transfer-in"
  | "planned-transfer-out"
  | "projected-fee"
  | "projected-interest"
  | "projected-refund"
  | "manual-projection";

export type ProjectionEntryStatus =
  | "planned"
  | "partially-funded"
  | "blocked"
  | "simulated";

export type ProjectionEntrySourceType =
  | "funding-source"
  | "debt-account"
  | "financial-obligation"
  | "transfer-plan"
  | "simulation"
  | "manual";

export type ProjectionEntry = {
  id: string;
  accountId: string;
  occurredOn: string;
  entryType: ProjectionEntryType;
  status: ProjectionEntryStatus;
  title: string;
  description?: string;
  amount: number;
  sourceType: ProjectionEntrySourceType;
  sourceId: string;
  sourceName?: string;
  priority: number;
  metadata?: Record<string, string | number | boolean | undefined>;
};
```

---

# 4. Field Semantics

## `id`

Unique deterministic identity for the projected movement.

The identifier should remain stable when the underlying planning inputs have not
changed.

Examples:

```text
funding-deposit-allocation-123
planned-settlement-credit-card-456-allocation-1
```

## `accountId`

The asset account whose projected balance changes.

For the initial implementation this must reference an active checking or savings
account.

## `occurredOn`

Canonical projected date in `YYYY-MM-DD` format.

## `entryType`

Describes the financial interpretation of the projected movement.

## `status`

Describes whether the movement is fully actionable, partially actionable,
blocked, or simulated.

Blocked entries should normally be surfaced as projection issues rather than
applied to running balances.

## `title`

User-facing primary label.

## `description`

Optional explanation of the movement.

## `amount`

Signed account-level movement.

Examples:

```text
+2500.00  deposit
-450.00   settlement
+200.00   transfer in
-200.00   transfer out
```

The sign must already reflect the effect on the target account.

## `sourceType`

Identifies the upstream domain that created the entry.

## `sourceId`

Identifies the originating domain record.

## `sourceName`

Optional human-readable source identity.

## `priority`

Deterministic same-day ordering.

Initial canonical priorities:

```text
0  opening position
10 funding deposit
20 transfer in
30 planned settlement
40 transfer out
50 projected fee
60 projected interest
90 manual projection
```

Opening position is not stored as a Projection Entry; it is supplied by the
account projection request.

## `metadata`

Optional structured explanation data.

Metadata must not contain core fields that belong in the canonical type.

# 5. Projection Issues

Movements that cannot safely become Projection Entries must be represented as
issues.

```ts
export type ProjectionIssueCode =
  | "missing-account"
  | "inactive-account"
  | "invalid-account-type"
  | "missing-routing"
  | "invalid-routing"
  | "blocked-funding-source"
  | "missing-projection-date"
  | "invalid-amount";

export type ProjectionIssue = {
  code: ProjectionIssueCode;
  sourceType: ProjectionEntrySourceType;
  sourceId: string;
  sourceName?: string;
  message: string;
  accountId?: string;
};
```

Projection issues must remain separate from valid entries.

The projection engine must never infer or repair routing silently.

# 6. Producer Responsibilities

## 6.1 Funding Deposit Projection

Input:

- Funding sources
- Funding deposit allocations
- Asset accounts

Output:

- Positive `funding-deposit` Projection Entries
- Projection issues for incomplete or invalid routing

Rules:

- Only planned funding sources participate
- Only fully conserved allocations participate
- Destination must be an active checking or savings account
- Entry date is the funding source availability date

## 6.2 Settlement Projection

Input:

- Operational Funding Plan
- Debt accounts
- Asset accounts

Output:

- Negative `planned-settlement` Projection Entries
- Projection issues for missing or invalid settlement routing

Rules:

- Only `debt-account` funding-plan items participate initially
- Zero allocations produce no entry
- Partial allocations project only the allocated amount
- Each funding allocation may produce its own settlement entry
- Entry date is the allocation payment date
- Target account is the debt account's `settlementAccountId`
- Utility obligations remain excluded until payment occurrences exist

## 6.3 Transfer Projection

Future capability.

A transfer must produce two entries:

```text
Source account  → planned-transfer-out
Destination     → planned-transfer-in
```

Both entries must reference the same transfer-plan source.

# 7. Account Projection Replay

The account projection engine consumes:

```ts
export type AccountProjectionRequest = {
  accounts: readonly FinancialAccount[];
  entries: readonly ProjectionEntry[];
};
```

For each active asset account:

```text
Opening Balance
+ Ordered Projection Entries
= Projected Closing Balance
```

The engine must calculate:

- Opening balance
- Total projected inflows
- Total projected outflows
- Lowest projected balance
- Closing projected balance
- Ordered entries with running balances

Proposed result:

```ts
export type AccountProjectionEntry = ProjectionEntry & {
  runningBalance: number;
};

export type AssetAccountProjection = {
  accountId: string;
  accountName: string;
  institutionName: string;
  accountType: "checking" | "savings";
  openingBalance: number;
  totalProjectedInflows: number;
  totalProjectedOutflows: number;
  lowestBalance: number;
  closingBalance: number;
  entries: AccountProjectionEntry[];
};
```

---

# 8. Deterministic Ordering

Projection Entries must be ordered by:

1. `occurredOn`
2. `priority`
3. `sourceId`
4. `id`

This creates stable replay and verification behavior.

For the same date:

```text
Funding Deposit
→ Transfer In
→ Planned Settlement
→ Transfer Out
→ Fee
→ Interest
```

Institution-specific cutoff timing may later require timestamp precision, but
date-only ordering remains canonical for the current operational model.

---

# 9. Projection Versus Ledger

Projection Entry:

```text
Expected or simulated future movement
```

Ledger Event:

```text
Posted financial history
```

A Projection Entry must never be appended directly to the operational ledger.

When actual activity occurs, a new immutable ledger event is recorded.

Later reconciliation may compare:

```text
Projection Entry
vs.
Ledger Event
```

Possible outcomes:

- Matched
- Amount variance
- Date variance
- Account variance
- Missing posted activity
- Unexpected posted activity

# 11. Architectural Decisions

## ADR-003.1.1 — Projection Entry is a planning primitive

Projection Entry is the canonical representation of future account movement.

Domain-specific engines produce entries; account projection replay consumes
them.

## ADR-003.1.2 — Signed amounts

Projection Entry amounts are signed from the perspective of the target account.

This avoids teaching the replay engine domain-specific debit and credit rules.

## ADR-003.1.3 — Routing remains explicit

Every Projection Entry must identify one target account.

Missing or invalid routing produces an issue, not an inferred entry.

## ADR-003.1.4 — Blocked movements do not affect balances

A blocked projected movement may be shown to the user, but it must not change
running balances.

## ADR-003.1.5 — Projection and history remain separate

Projection Entries never become ledger events automatically.

Posted history requires an explicit immutable financial event.

---

# 12. Summary

The Projection Entry model creates a stable boundary between financial planning
domains and account-level replay.

Canonical architecture:

```text
Domain Planning Engines
→ Projection Entries
→ Account Projection Replay
→ Liquidity State
→ Recommendations
```

This allows future recommendations such as:

```text
Transfer $200 to Primary Checking before July 23
```

to be derived from explicit, ordered, explainable account movements rather than
from aggregate cash assumptions.
